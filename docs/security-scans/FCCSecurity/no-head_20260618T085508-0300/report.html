[1] OpenAI-facing wording overstates no dependency install/static-only posture while repo contains optional Python PDF tooling with third-party imports

FieldValue
SeverityMEDIUM
PriorityP2
CWEnone
Affected linesdocs/openai/openai_deploy_review_request.md:29-30; docs/openai/openai_developer_forum_post.md:28,30; docs/threat-model/threat_model.md:48; scripts/generate_profile_pdfs.py:8-17,31

The OpenAI-facing and threat-model wording describes the project as static HTML/CSS/JavaScript only and says there is no dependency install. That is accurate for the deployable dashboard runtime, but the repository also contains optional local PDF tooling in scripts/generate_profile_pdfs.py that imports PIL, pypdf, reportlab, and optionally pypdfium2.

Validation used full-file line-numbered static review, targeted rg, and repo-state checks. No package-manager manifest was found, and no runtime unsafe DOM sink, secret exposure, external deploy, or OpenAI affiliation/approval claim was observed.

Severity is MEDIUM because the issue crosses the documentation/evidence boundary defined by the threat model: reviewers may consume these docs as public or OpenAI-facing evidence and infer a broader no-dependency/static-only claim than the repository supports.

Recommended fix: scope no-dependency/static-only wording to the deployable dashboard runtime and document optional local Python PDF tooling separately.
