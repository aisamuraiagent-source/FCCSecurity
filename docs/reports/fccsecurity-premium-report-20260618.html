1. Sumário Executivo

O caso começou como bloqueio persistente de escrita no projeto FCCSecurity dentro do OneDrive. A análise demonstrou que o problema prático não era falta simples de ACL, mas interferência de proteção de pasta controlada e do fluxo protegido do OneDrive. A solução correta foi preservar evidência, criar backup integral, reconstruir um workspace limpo em C:\tmp e consolidar o projeto como aplicação local-first auditável.

Resultado: projeto local completo, validado, empacotado, sem GitHub ativo, sem deploy, sem instalação de dependências, sem commit/push e sem alteração em Defender, OneDrive, ACL ou permissões do sistema.

2. Insights Que Resolveram o Caso

#InsightEfeito prático
1Separar problema local de problema de nuvem.Evita deletar ou publicar errado. O GitHub não era o bloqueio de escrita; o bloqueio estava no caminho OneDrive.
2ACL permissiva não prova capacidade real de escrita.A camada de Controlled Folder Access/OneDrive podia bloquear mesmo com permissões aparentes corretas.
3Desvio operacional seguro é melhor que abrir permissões amplas.Usar C:\tmp preservou segurança do Windows e recuperou produtividade sem reduzir controle.
4Backup antes de reconstrução.Preservou evidência e reduziu risco de perda de contexto técnico.
5Clean tree precisa carregar só o que é ativo.Artefatos brutos antigos e referências de scan/deploy foram removidos do fluxo limpo.
6Validação reexecutável vale mais que validação narrativa.scripts\validate-local.js virou gate local sem dependências.
7Frontend local também precisa contrato funcional.Views Board, Evidence e Validate deixaram de ser apenas botões ativos e passaram a controlar painéis reais.
8Notas locais são evidência, não cofre.Foram criados limite de tamanho, wrappers seguros e metadados explícitos de exportação.
